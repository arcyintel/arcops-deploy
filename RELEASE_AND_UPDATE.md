# ArcOps — Release & Update System

> Zero-touch, versioned, atomic fleet updates — **images AND the host files under
> `/opt/arcops`** (docker-compose, `.env`, Caddyfile, mosquitto.conf) — with
> backup, health-gating and rollback. Replaces the old "watchtower watches
> `:latest` + scp the compose by hand" model that froze every box's host files
> at install time.

Companion to `DEPLOYMENT.md` (first-time install) and `ARCHITECTURE.md`.

---

## 1. The model in one picture

```
                         push to main
                            │  each service CI → ghcr :latest (+ :sha)
                            ▼
   ┌─────────────────────────────────────────────┐
   │ TEST SERVER  (test.uconos.com)               │  ARCOPS_CHANNEL=edge
   │ arcops-updater (edge mode)                   │  → tracks :latest + main host files
   │  • polls every ~120s                         │  → applies when an image digest OR
   │  • change? backup → apply → health → rollback│    a host file actually changed
   └───────────────┬─────────────────────────────┘  (else idle, exit 5)
                   │  validate here
                   ▼
        git tag commons vX.Y.Z   ──►  release.yml
                                       imagetools re-tag every :latest → :X.Y.Z
                                       (no rebuild; the validated digest is frozen)
                   │
                   ▼
        promote.yml (manual "ship it")
          • imagetools re-tag every :X.Y.Z → :stable
          • write + commit release/manifest-stable.json
          • POST it to the licence-server  (GET /api/v1/release/manifest)
                   │
                   ▼
   ┌─────────────────────────────────────────────┐
   │ CUSTOMER FLEET                               │  ARCOPS_CHANNEL=stable
   │ arcops-updater (stable mode)                 │  → reads the manifest hourly
   │  • manifest version > installed?             │  → pins *_TAG=X.Y.Z in .env,
   │  • backup → fetch vX.Y.Z host files →        │    fetches that tag's host files,
   │    .env merge → pull → up → health → rollback│    applies atomically
   └─────────────────────────────────────────────┘
```

**Tag meaning:** `:latest` = dev/edge (moving), `:X.Y.Z` = a cut release (frozen),
`:stable` = what customers run (moving pointer at the promoted `:X.Y.Z`).

## 2. Channels

| | **edge** (test/staging) | **stable** (customers) |
|---|---|---|
| `ARCOPS_CHANNEL` | `edge` | `stable` |
| Tracks | `:latest` + `commons/main` host files | manifest's `:X.Y.Z` + `commons/vX.Y.Z` host files |
| Trigger | any image digest / host-file change | manifest `version` > installed (semver) |
| Poll | ~120 s | 3600 s |
| Manifest | none (derives from main) | `GET /api/v1/release/manifest?channel=stable` |
| Set up with | `setup.sh --channel edge` | `setup.sh` (default) |

Both run the **same** `arcops-updater` image and the **same** `arcops-update.sh`
core — only `.env` differs. Edge is where you validate a build (images + compose
+ config) **before** promoting it to the fleet.

## 3. Components (what lives where)

**commons** (the source of truth for host-side machinery)
- `src/main/resources/arcops-update.sh` — the 7-step core: preflight → resolve
  (manifest / edge-from-main / `--bundle`) → backup (host files + `pg_dump` per
  schema + previous tags) → `.env` merge (additive; never overwrites secrets) +
  pin `*_TAG` + `INSTALLED_VERSION` → stage host files → `compose pull && up -d`
  (excluding the updater itself) → health-gate → rollback on failure. Exit codes:
  `0` applied · `5` nothing-to-do · `10` preflight · `20` fetch · `30` apply · `40` health(rolled back).
- `updater/Dockerfile` + `updater/entrypoint.sh` — the `arcops-updater` image
  (docker-cli + compose + the script). entrypoint loops, and on `rc=0` runs the
  **self-update hand-off** (a transient `--rm` helper recreates the updater with
  its new image so it never kills itself mid-apply).
- `src/main/resources/.env.template` — every env key + default (the merge source).
- `src/main/resources/release/manifest-stable.json` — committed placeholder;
  regenerated by `promote.yml`.
- `src/main/resources/arcops-export-release.sh` — air-gapped bundle producer
  (`docker save` all `:X.Y.Z` + host files + manifest → `arcops-release-X.Y.Z.tar.gz`).
- `src/main/resources/docker-compose-production.yaml` — `arcops-updater` service
  (watchtower removed), `*_TAG` default `stable`, `INSTALLED_VERSION` passed to gateway.
- `src/main/resources/setup.sh` — `--channel`, `--version`, version-pinned host-file
  fetch, `.env` with `INSTALLED_VERSION` + updater knobs + derived `ARCOPS_MANIFEST_URL`.
- `.github/workflows/updater-image.yml` — builds `arcops-updater:latest` on main.
- `.github/workflows/release.yml` — `vX.Y.Z` tag ⇒ re-tag every service `:latest → :X.Y.Z`.
- `.github/workflows/promote.yml` — manual ⇒ re-tag `:X.Y.Z → :stable` + write/commit
  the manifest + POST it to the licence server.

**license-server** (the fleet-facing endpoint + visibility)
- `ReleaseController` — `GET /api/v1/release/manifest` (public; what the fleet reads)
  + `POST /api/v1/release/manifest` (guarded by `X-Release-Token`; promote.yml publishes).
- `ReleaseManifest` entity + `ReleaseManifestService` + `V10` migration (`release_manifest`
  table + `licenses.installed_version` / `last_seen_at`).
- `LicenseCheckController` — captures `X-Installed-Version` on the existing hourly,
  authenticated `/license/check` (throttled ≤ every 30 min). No separate phone-home.
- `FleetController` / `FleetService` — `GET /portal/api/fleet` (staff): who runs what,
  how many are behind stable.

**gateway** — sends `X-Installed-Version: ${INSTALLED_VERSION}` on its hourly licence check.

**licence-portal** — `Fleet` page (`/licence/fleet`, "Filo" in the nav): per-customer
version, up-to-date / behind, last-seen.

## 4. Operating it

### Day-to-day (automatic)
Push to `main` → each service publishes `:latest` → the **test server** (edge)
picks it up within ~2 min (images + any compose/config change), backs up,
health-gates, rolls back on failure. Nothing to do by hand.

### Cut a release
```
# in the commons repo, once main is validated on the edge box:
git tag v1.5.0 && git push origin v1.5.0
```
`release.yml` re-tags every service's current `:latest` digest → `:1.5.0`. The
`vX.Y.Z` tag also freezes the host files the manifest will point at.

### Promote to the fleet
Run **promote.yml** (`workflow_dispatch`) with `version=1.5.0`
(optional `minFromVersion`, `notes`). It re-tags `:1.5.0 → :stable`, writes +
commits `manifest-stable.json`, and POSTs it to the licence server. Stable boxes
converge on their next hourly poll.

> Promotion is intentionally **manual** — it is the "ship to customers" gate.
> Auto-promote-on-green can be added later.

### Install a box
```
# customer (stable, versioned):
sudo ./setup.sh --domain mdm.acme.com --license-server-url https://licence.arcyintel.com
# test/staging (edge, tracks latest):
sudo ./setup.sh --domain test.uconos.com --license-server-url https://licence... --channel edge
# pin a specific release:
sudo ./setup.sh --domain mdm.acme.com --version 1.5.0 --license-server-url https://licence...
```

### Air-gapped customer
```
# on an online machine:
./arcops-export-release.sh --version 1.5.0 --manifest-url https://licence.arcyintel.com/api/v1/release/manifest
# hand-carry arcops-release-1.5.0.tar.gz to the box, then:
sudo ./arcops-update.sh --bundle arcops-release-1.5.0.tar.gz
```

### Rollback
Automatic: if the health gate fails after an apply, the updater restores the
previous `.env` + compose + Caddyfile + mosquitto and brings the prior images
back up. DB is **not** auto-restored (per-schema `pg_dump` snapshots are kept in
`/opt/arcops/backups/<ts>/`) — rely on expand-contract migrations so a
rolled-back image still runs against the new schema.

## 5. One-time setup (required before the pipeline works)

**commons repo → Settings → Secrets → Actions**
- `GHCR_PAT` — a token with `write:packages` over the `arcyintel/*` image
  packages. Needed by release.yml + promote.yml to re-tag images that the
  sibling service repos own (the default `GITHUB_TOKEN` can't).
- `LICENCE_SERVER_URL` — e.g. `https://licence.arcyintel.com` (promote.yml live publish).
- `ARCOPS_RELEASE_PUBLISH_TOKEN` — shared secret matching the licence server's env.

**licence-server deployment env**
- `ARCOPS_RELEASE_PUBLISH_TOKEN` — same value as the commons secret; authorizes
  `POST /api/v1/release/manifest`. Empty ⇒ publishing disabled (GET still serves
  the last-published manifest).

**customer/box `.env`** (written by setup.sh; shown for reference)
- `ARCOPS_CHANNEL` (`stable`/`edge`), `ARCOPS_MANIFEST_URL`
  (`<licence>/api/v1/release/manifest`), `INSTALLED_VERSION`, `*_TAG`,
  `ARCOPS_AUTO_UPDATE`, `ARCOPS_UPDATE_POLL_INTERVAL`, `ARCOPS_UPDATER_TAG`.

## 6. Cutover runbook — test.uconos.com (watchtower → arcops-updater)

One-time, per existing box (only the handful that predate this system).

1. **Pull the new compose + .env keys** onto the box (first time, by hand — after
   this the updater self-maintains):
   ```
   cd /opt/arcops
   cp docker-compose.yaml docker-compose.yaml.bak && cp .env .env.bak
   curl -fsSL https://raw.githubusercontent.com/arcyintel/arcops-deploy/main/src/main/resources/docker-compose-production.yaml -o docker-compose.yaml
   ```
2. **Add the updater knobs** to `.env` (edge box):
   ```
   ARCOPS_CHANNEL=edge
   ARCOPS_AUTO_UPDATE=true
   ARCOPS_UPDATE_POLL_INTERVAL=120
   ARCOPS_UPDATER_TAG=latest
   ARCOPS_EDGE_TAG=latest
   ARCOPS_EDGE_BASE_URL=https://raw.githubusercontent.com/arcyintel/arcops-deploy/main
   ARCOPS_EDGE_CADDYFILE_URL=        # leave empty if this box runs a licence-profile Caddyfile
   INSTALLED_VERSION=edge
   ```
3. **Stop + remove watchtower, start the updater:**
   ```
   docker compose stop watchtower && docker rm -f watchtower
   docker compose pull arcops-updater
   docker compose up -d arcops-updater
   docker logs -f arcops-updater    # watch one cycle
   ```
4. Confirm a normal cycle logs `edge: nothing new — up to date` (or an apply +
   `all services healthy`). The box is now self-maintaining.

> ⚠️ If the test box runs the licence profile with a **custom Caddyfile**, keep
> `ARCOPS_EDGE_CADDYFILE_URL` empty so edge never clobbers it (compose + mosquitto
> + `.env` are still tracked).

## 7. Roll-forward discipline + PR checklist

The fleet updates **forward**; structural host-file changes only reach boxes on a
promote. So:

- **New env var?** Ship it with an image-side default
  (`@Value("${X:default}")` / compose `${X:-default}`) **and** add it to
  `.env.template`. The updater appends missing keys with their template value;
  it never rewrites existing (secret) values. A var with no default that isn't in
  `.env` will break boxes that haven't merged it yet.
- **New service / volume?** Add it to `docker-compose-production.yaml`. It reaches
  boxes zero-touch on the next promote (the updater runs `up -d` with the new compose).
- **Breaking DB migration?** Use expand-contract (add nullable → backfill →
  switch → drop in a later release) so a rolled-back image still runs. Set
  `minFromVersion` on the promote if a stepping-stone release is mandatory.
- **mqtt / Caddy change?** Edit `mosquitto.conf` / `Caddyfile.no-licence` in
  commons; bump the checksums flow handles the rest (promote recomputes them).

**PR checklist (paste into PRs that touch deploy surface):**
- [ ] New env vars have an image default **and** are in `.env.template`.
- [ ] Compose changes are backward-safe (a rolled-back image still boots).
- [ ] DB migrations are expand-contract (no destructive change in the same release).
- [ ] If a stepping-stone is required, `minFromVersion` is set on promote.
- [ ] Health endpoints still pass for every service the gate checks
      (gateway, back-core, identity, apple-mdm, android-mdm, windows-mdm).

## 8. Notes / deferred

- **Version embedding (`/actuator/info`)** — deferred. The authoritative "what
  version is each box on" is `INSTALLED_VERSION` in `.env`, which the gateway
  reports to the Fleet view on every licence check. Baking the release version
  into each image via a build-arg (for `/actuator/info` cross-check + an FE
  footer) is an optional future enhancement; it is **not** needed for updates or
  fleet visibility, which already work without it.
- **Updater self-update on stable** — `ARCOPS_UPDATER_TAG=stable` is a moving tag;
  `compose pull` fetches the promoted updater image, then the hand-off recreates
  it. On edge the tag is `latest`.
- **docker.sock exposure** — the updater holds the same host-root access
  watchtower did; the image is minimal + pinned.
- **Licence server's own box** — it is Arcyintel infra, not part of the customer
  fleet manifest; update it with its own mechanism (it is not driven by `arcops-updater`).
